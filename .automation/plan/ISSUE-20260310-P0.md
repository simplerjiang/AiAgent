# ISSUE-20260310-P0 Plan - Dynamic Source Governance (Native AI)

## Goal
在“来源经常变化”的前提下，建立可自我维护的资讯源治理系统：LLM 每日自动发现候选源，程序自动验证与分级，尽量减少人工干预。

## Why P0
- 若来源治理不先做，P1/P2 的抗污染与新闻库都会受到输入质量上限限制。
- P0 是数据入口质量基线，优先级高于单一策略优化。

## Native-AI Architecture
1. Source Registry（来源注册表）
   - 保存来源元数据：domain、type、tier、fetchStrategy、parserVersion、status、qualityScore、lastSuccessAt。
2. LLM Discovery Agent（每日发现）
   - 每日扫描并提出候选来源（含推荐理由、覆盖主题、预期可信级别）。
3. Auto Verification Pipeline（自动准入）
   - 自动抓样、结构校验、时间戳校验、重复率/新鲜度/可解析率评估。
4. Auto Promotion / Quarantine（自动升降级）
   - 达标自动进入 fallback/preferred；异常自动隔离并停止生产使用。
5. Analyst LLM Consumption（分析侧使用）
   - 分析 Agent 只消费“已通过治理”的来源，不直接消费未验证候选源。
6. LLM Crawler Maintainer（自动爬虫维护）
   - 根据失败日志自动生成/更新站点抓取配置与解析器代码补丁。
   - 变更先在沙箱验证并跑测试，通过后自动合并到运行版本。

## Automation-First Policy
- 默认无人工模式：发现、验证、分级、隔离、恢复均自动进行。
- 人工仅处理高影响异常（例如 authoritative 源连续失败、误判激增）。

## Work Breakdown
1. 数据模型
   - `source_registry`
   - `source_health_daily`
   - `source_candidate_queue`
   - `source_verification_runs`
   - `crawler_change_queue`（LLM 生成的爬虫改动任务）
   - `crawler_change_runs`（改动验证与发布记录）
2. 每日任务编排
   - `discover-sources`（LLM）
   - `verify-candidates`（程序）
   - `re-score-sources`（程序）
   - `promote-or-quarantine`（规则引擎）
   - `auto-fix-crawlers`（LLM 读取失败样本后生成补丁）
   - `validate-crawler-patches`（沙箱回放 + 单测 + 安全检查）
   - `deploy-crawler-patches`（仅发布通过验证的补丁）
3. 评分体系（示例）
   - parseSuccessRate、freshnessLag、duplicateRate、timestampCoverage、contentDepth、crossSourceAgreement
4. 准入阈值（示例）
   - `preferred`: parseSuccessRate >= 95%, timestampCoverage >= 98%, freshnessLag <= 30 min
   - `fallback`: parseSuccessRate >= 85%, timestampCoverage >= 90%
   - 否则 `quarantine`
5. 供给约束
   - P1/P2 分析链路仅允许使用 `authoritative/preferred/fallback` 且必须有时间戳。

## Autonomous Crawler Policy
1. LLM 自动新增来源地址
   - LLM 每日输出候选地址列表（domain + rss/api/page + 抓取方式建议）。
   - 程序自动执行连通性与样本抓取验证，不通过则不入库。
2. LLM 自动生成爬取流程
   - 输出 `fetchStrategy`（rss/html/api/headless）、提取规则、分页策略、反爬处理策略。
   - 统一转为结构化配置（JSON）后执行，避免直接执行自由文本指令。
3. LLM 自动更新爬虫代码
   - 触发条件：连续失败、结构漂移、时间戳解析失败率超阈值。
   - LLM 生成补丁后，必须经过：
     - 编译检查
     - 目标解析器定向单测
     - 历史样本回放（成功率/字段完整性）
     - 安全策略检查（仅允许白名单目录变更）
4. 自动回滚
   - 新补丁上线后若 24h 健康分下降到阈值以下，自动回滚上一稳定版本并标记 quarantined。

## Safety Guardrails (No-Human-by-Default)
- 默认自动执行，但加入硬性闸门：
  - 仅允许修改 `backend/.../Modules/Stocks/Services/*Parser*.cs` 和对应测试文件。
  - 禁止自动改动认证、数据库连接、安全配置等高风险文件。
  - 每次自动改动必须附带 traceId、原因、测试结果、回滚点。
- 人工介入仅在两类情况：
  - 连续 N 次自动修复失败
  - authoritative 来源整体不可用

## Acceptance Criteria
- 每日自动发现候选来源并完成自动验证，无需人工触发。
- 来源状态自动变化可追溯（who/when/why/score）。
- 当来源异常波动时，系统自动降级或隔离，分析链路不中断。
- 仅在高影响异常场景触发人工告警。
- 当爬虫失效时，LLM 可自动提交修复补丁并经测试后自动发布。
- 自动发布补丁支持健康回滚，确保连续可用性。

## Tests (Required Order)
1. 单元测试：评分、阈值、状态迁移、隔离恢复逻辑。
2. 集成测试：发现 -> 验证 -> 升降级 -> 分析消费全链路。
3. 若新增 UI 面板，再做 Edge MCP 交互与日志检查。

## Deliverables
- P0 任务入队与文档同步（README/tasks/report/state）
- P2 计划升级：显式依赖 P0 来源治理基座
- 自动爬虫维护流水线设计（发现 -> 生成补丁 -> 校验 -> 发布 -> 回滚）
