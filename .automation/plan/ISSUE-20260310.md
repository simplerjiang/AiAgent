# ISSUE-20260310 Plan - Phased Execution Blueprint

## Objective
把周记中的三个复杂问题拆成可并行规划、可串行交付、可验证回归的实施路线。

## Phase Structure
- P0-R1: Frontend developer mode visual replay for source governance operations
- P1: News anti-pollution guardrails for all Agent/LLM news usage
- P2: Internal news knowledge base + scheduled ingestion/analysis loop
- P3: White-box MCP/Skill framework + scheduled unfinished-task execution

## P0-R1 - Frontend Developer Mode Visual Replay

### Scope
- 在前端新增 Developer Mode，提供来源治理状态可视化复现能力。
- 提供治理仪表盘：来源健康、候选源验证、修复队列、回滚历史、traceId 查询。
- 后端补最小只读查询接口，保持权限边界与审计可追踪。

### Implementation Steps
1. 定义前端可视化所需只读 DTO 与分页查询契约。
2. 后端新增 Developer Mode 查询接口并补齐单测。
3. 前端新增开发者模式页面，支持筛选、展开详情、trace 跳转。
4. 增加错误快照与 replay 结果展示组件。

### Acceptance Criteria
- 开发者模式可开关，且普通模式不暴露治理细节。
- 仪表盘可稳定渲染且核心交互可用（筛选/展开/trace 跳转）。
- 后端与前端日志无运行时错误。

### Tests
- Unit tests first (backend + frontend).
- Then Edge MCP interaction checks with backend log and frontend console inspection.

## P1 - News Anti-Pollution Guardrails

### Scope
- 统一来源分层：authoritative / preferred / fallback / blocked。
- 统一时效门槛：默认 72h，证据不足可扩窗 7d 且必须显式标注。
- 统一降级逻辑：无来源或无时间戳关键证据时，强制 neutral/insufficient-data。

### Implementation Steps
1. 在后端新闻上下文组装层增加 source-tier 过滤与打分。
2. 在多 Agent 汇总与 Commander 改判链路增加“证据质量闸门”。
3. 在提示词模板中固化“证据不足降级”的硬规则。
4. 在输出结构中增加 evidenceQualitySummary 字段。

### Acceptance Criteria
- 低质量来源不会驱动高置信方向性建议。
- 所有关键证据都具备来源+发布时间，否则结论自动降级。
- 单测覆盖：blocked-only、untimestamped-only、mixed-source 三类场景。

### Tests
- Backend unit tests (targeted first, then full suite).
- 若涉及前端展示字段，再执行 Edge MCP 交互与日志检查。

## P2 - News Knowledge Base + Scheduler

### Scope
- 建立 market/sector/stock 三层新闻库。
- 数据入库带去重键、来源等级、发布时间、抓取时间、主题标签。
- 定时任务周期化拉取并触发 LLM 结构化摘要。

### Implementation Steps
1. 设计并落地新闻核心表与索引（news_items/news_tags/news_entities/news_runs）。
2. 实现采集管线（源抓取 -> 清洗 -> 去重 -> 入库）。
3. 实现调度任务（固定频率 + retry/backoff + dead-letter）。
4. 提供按 symbol/sector/time-window 的查询接口供 Agent 调用。

### Acceptance Criteria
- 重复新闻不重复入库；同一 URL/标题哈希命中去重。
- 调度失败可重试并有审计记录，任务状态可追踪。
- Agent 可读取近 72h 新闻上下文并带来源等级。

### Tests
- SQL schema verification with sqlcmd (tables/columns/indexes).
- Backend ingestion + dedupe unit tests.
- Scheduler run simulation tests.

## P3 - White-box MCP/Skill + Unfinished Task Loop

### Scope
- MCP/Skill 能力注册白盒化（输入/输出/权限/副作用可见）。
- 未完成任务支持定时扫描、可恢复执行、状态机可观测。
- 审计链路统一 traceId。

### Implementation Steps
1. 增加 Capability Registry（name, contract, permission, owner, version）。
2. 增加 Task Runtime（pending/running/succeeded/failed/cancelled）。
3. 实现 Cron/interval 扫描器执行未完成任务并写入审计日志。
4. 暴露最小可视化查询接口（任务状态、最近错误、重试次数）。

### Acceptance Criteria
- 每个 Skill 调用都可追踪权限与执行结果。
- 未完成任务可自动继续执行，不会无限重试失控。
- 审计日志可按 traceId 串联请求-执行-失败/成功。

### Tests
- Backend deterministic tests for permission boundary + state transitions.
- 若新增前端任务面板，再执行 Edge MCP 关键交互验证。

## Cross-Phase Rules
1. 测试顺序固定：Unit -> Edge MCP（仅 UI 改动时）。
2. 每阶段结束更新 `.automation/tasks.json`、`.automation/state.json`、`README.md`。
3. 每阶段产出双语报告（EN + ZH），记录命令、结果、问题与修复。
4. 若阶段引入数据库变更，先 sqlcmd 本地验证结构正确性。

## Delivery Order (Recommended)
1. P0-R1 first (operational visibility baseline)
2. P1 second (risk control baseline)
3. P2 third (data foundation)
4. P3 fourth (execution automation)

## Rollback Strategy
- 每个 Phase 独立提交，失败时按阶段回滚。
- 对调度器与任务运行时变更采用 feature flag，便于灰度启用。
