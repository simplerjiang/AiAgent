# ISSUE-20260310-P0-R1 Plan - Frontend Developer Mode Visual Replay

## Objective
Upgrade the remaining P0 scope into an executable plan for a pure-frontend Developer Mode that can visually replay source-governance runs with deterministic observability.

## Why This Is P0-Remaining
- P0 backend governance loop is already closed and tested.
- Current gap is operator visibility and reproducible debugging from frontend.
- This work is required to make P0 operationally usable, not just technically complete.

## Scope
- Frontend Developer Mode entry and guardrail
- Governance dashboard for current/recent system state
- Visual replay and error snapshot inspection
- Minimal backend read-only query APIs for the dashboard
- Trace-linked observability for queue/run/rollback views

## Non-Goals
- No new autonomous mutation capability in frontend
- No bypass of existing backend permission boundaries
- No replacement of P1/P2/P3 long-term architecture

## Deliverables
1. Developer Mode toggle
- Add a clear Developer Mode switch in frontend settings or admin panel.
- Hide advanced governance data in normal mode.

2. Governance Dashboard (frontend)
- Source health summary cards (active/quarantined/new candidates).
- Candidate source list with verification outcome and score breakdown.
- Crawler change queue panel with lifecycle states.
- Rollback history panel with reason and result.
- Trace search box (`traceId`) with jump links to related records.

3. Replay & Snapshot Panel (frontend)
- Display proposal payload summary (patch count, target files, replay command).
- Show validation gate results (build/test/replay) with timestamps.
- Provide read-only replay metadata view and error snapshot viewer.

4. Minimal Backend APIs
- Add read-only endpoints for dashboard widgets and tables.
- Keep contracts stable and paginated.
- Return strict DTOs with required timestamps and status enums.

5. Audit and Safety
- Every frontend card row links to `traceId`.
- Sensitive fields redacted in API responses.
- Enforce role-based access for Developer Mode endpoints.

## Acceptance Criteria
- Developer Mode can be enabled and disabled from frontend.
- Dashboard loads data without backend errors.
- Key interactions are usable: filter, detail expand, trace jump.
- Rollback and validation history are visible and consistent with backend state.
- Edge MCP run confirms rendering + interactions + clean console/backend logs.

## Implementation Steps
1. Define API contract DTOs and response models for visual needs.
2. Implement backend read-only endpoints and unit tests.
3. Build frontend Developer Mode page and reusable data widgets.
4. Wire trace-based navigation and detail drawers.
5. Add focused frontend tests for state mapping and rendering.
6. Run Unit tests first, then Edge MCP validation.

## Tests (Required Order)
1. Backend unit tests for new read-only endpoints and status mapping.
2. Frontend unit tests for dashboard components and trace navigation state.
3. Edge MCP checks for visual render, interaction, and log cleanliness.

## Risks and Mitigations
- Risk: UI data density causes confusion.
- Mitigation: Use compact summary + drill-down details.

- Risk: Trace linking mismatch across entities.
- Mitigation: Add deterministic trace join tests in backend.

- Risk: Unauthorized access to diagnostics.
- Mitigation: Explicit admin/developer authorization policy.

## Exit Definition
P0-R1 is complete only when frontend Developer Mode can reproduce governance lifecycle state visually and all required tests pass in the mandated order.
