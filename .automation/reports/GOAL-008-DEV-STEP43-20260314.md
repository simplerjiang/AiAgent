# GOAL-008 Step 4.3 Development Report (2026-03-14)

## EN
### Summary
Completed the Step 4.3 rework of the trading-plan engine. The backend now evaluates pending plans from local cached quote/minute data only within the enabled `ActiveWatchlist`, warning events are deduplicated by an explicit ongoing-condition window key instead of raw JSON equality, and the frontend keeps both the global trading-plan board and the current-stock plan card in the short-poll refresh path.

### Implementation
- Backend:
  - Added `TradingPlanEvent` with tolerant enum conversion and EF mapping in `AppDbContext`.
  - Added additive schema support plus migration for `TradingPlanEvents`.
  - Added `TradingPlanTriggerOptions`, `TradingPlanTriggerWorker`, and `TradingPlanTriggerService`.
  - Added `GET /api/stocks/plans/alerts` in `StocksModule` and a new `TradingPlanEventItemDto`.
  - Implemented deterministic state transitions:
    - `Pending -> Invalid` when invalid condition matches
    - `Pending -> Triggered` when trigger condition matches
    - `Invalid` wins when both match
  - Added warning-only recent minute volume-divergence event emission without changing plan status.
  - Reworked `TradingPlanTriggerService` so Step 4.3 now gates evaluation by enabled `ActiveWatchlist` symbols instead of scanning all `Pending` plans.
  - Reworked warning idempotency so repeated passes compare a normalized `warningWindowKey` carried in event metadata, preventing the same ongoing divergence from writing duplicate warnings every poll.
- Frontend:
  - Added plan-alert workspace state, request tokens, abort handling, and `fetchTradingPlanAlerts(...)`.
  - Added `refreshTradingPlanBoard(...)` / `refreshTradingPlanSection(...)` so plan list and alert summary refresh together.
  - Added alert summary rendering to the global board and current-plan card.
  - Added short polling for the active stock plan section and cleaned up the new abort controller/timer paths on unmount.
  - Reworked `setupPlanRefresh()` so the same timer also refreshes the global trading-plan board even when no stock is active.
  - Added alert summary styling.
- Test updates:
  - Added `TradingPlanTriggerServiceTests.cs` for trigger, invalid, priority, idempotency, and divergence warning coverage.
  - Extended `StockInfoTab.spec.js` with alert-summary rendering/refresh coverage and updated shared mocks for the new alerts endpoint.
  - Added explicit regression coverage for skipping plans outside `ActiveWatchlist`, deduplicating divergence warnings within the same window, and polling the trading-plan board when no stock is active.

## ZH
### 摘要
已完成交易计划引擎 Step 4.3 的返工收口。后端现在只在启用的 `ActiveWatchlist` 范围内，基于本地 quote/minute 缓存评估 `Pending` 计划；量价背离 warning 改为按持续条件时间窗去重；前端则通过短轮询同时刷新“交易计划总览”和“当前交易计划”卡片。

### 实现内容
- 后端：
  - 新增 `TradingPlanEvent`，并在 `AppDbContext` 中补齐宽容枚举转换与 EF 映射。
  - 新增 `TradingPlanEvents` 的 additive schema 初始化与 migration。
  - 新增 `TradingPlanTriggerOptions`、`TradingPlanTriggerWorker`、`TradingPlanTriggerService`。
  - 在 `StocksModule` 中新增 `GET /api/stocks/plans/alerts` 与 `TradingPlanEventItemDto`。
  - 实现确定性状态机：
    - 命中失效条件时 `Pending -> Invalid`
    - 命中触发条件时 `Pending -> Triggered`
    - 同轮同时命中时 `Invalid` 优先
  - 增加最近分钟量价背离 `Warning` 事件，但不直接改变计划状态。
  - 返工 `TradingPlanTriggerService`，让 Step 4.3 只处理启用 `ActiveWatchlist` 白名单内的计划，不再绕开白名单扫描所有 `Pending` 计划。
  - 返工 warning 幂等逻辑，在事件 metadata 中引入归一化 `warningWindowKey`，用“同一持续条件时间窗”替代原先的原始 JSON 字符串相等去重。
- 前端：
  - 新增 plan alert 的 workspace 状态、request token、abort 处理与 `fetchTradingPlanAlerts(...)`。
  - 新增 `refreshTradingPlanBoard(...)` / `refreshTradingPlanSection(...)`，让计划列表与告警摘要一起刷新。
  - 在跨股票总览卡和当前股票计划卡中展示最新告警摘要。
  - 为当前激活股票增加计划短轮询，并在组件卸载时清理新的 abort controller 与 timer。
  - 返工 `setupPlanRefresh()`，让同一个定时器在无选股时也持续刷新跨股票总览，在有选股时同时刷新总览与当前股票计划区。
  - 新增告警摘要样式。
- 测试补充：
  - 新增 `TradingPlanTriggerServiceTests.cs`，覆盖触发、失效、优先级、幂等和背离 warning。
  - 扩展 `StockInfoTab.spec.js`，增加告警摘要渲染/刷新用例，并同步更新共享 mock，适配新的 alerts 接口。
  - 显式补充回归：白名单外计划不执行、同一时间窗内背离 warning 不重复落库、无选股时交易计划总览仍会轮询。
