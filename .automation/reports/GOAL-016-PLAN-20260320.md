# GOAL-016 Plan / 单机可安装版与本地数据底座重构规划

## Plan (EN)
- Objective: restructure the current development-oriented shape into a Windows application that can be installed and used on different machines without requiring users to install SQL Server or manually start frontend and backend processes.
- Target architecture:
  - desktop host: WinForms + WebView2 executable acts as the application host
  - backend: ASP.NET Core API starts locally under desktop control
  - frontend: prebuilt static assets ship with the application package
  - transactional store: SQLite for configuration, watchlists, trading plans, hot-cache data, and other application-state tables
  - archive store: Parquet partitions for large historical minute/K-line/news/history payloads
  - analytics engine: DuckDB for local scans, replay, backtesting, and aggregate queries over Parquet
- Why this direction:
  - current code is tightly coupled to SQL Server startup SQL and schema repair logic
  - future large-scale growth makes a single ever-growing embedded transactional database a poor long-term fit
  - the product goal is local-first distribution, not workstation-specific environment setup

## 计划（ZH）
- 目标：把当前偏开发态的“桌面壳 + 本地后端 + 外部 SQL Server”形态，改造成可以发给不同 Windows 用户安装使用的单机应用，不要求用户安装 SQL Server，也不要求手工启动多个进程。
- 目标架构：
  - 桌面宿主：WinForms + WebView2 EXE 作为应用宿主
  - 后端：ASP.NET Core API 由桌面程序本地拉起并托管
  - 前端：预构建静态资源随应用一起发布
  - 主事务库：SQLite，承载配置、观察列表、交易计划、热数据缓存和其他应用状态表
  - 冷归档：Parquet 分区文件，承载大体量历史分时、K线、新闻/事实归档和分析历史
  - 本地分析引擎：DuckDB，负责对 Parquet 做回放、回测、聚合和批量统计
- 这样设计的原因：
  - 当前代码对 SQL Server 启动期补表/补列逻辑耦合较重
  - 按未来大数据量设计时，把所有长期历史都堆进一个嵌入式事务库不是长期正确方案
  - 项目目标是本地优先分发，而不是让每台机器单独搭建数据库环境

## Phase Checklist (EN)
- R1 Provider abstraction and SQLite foundation
  - formalize provider selection and local data-path configuration
  - remove SQL Server-only startup schema logic
  - make the existing core schema and migrations run on SQLite
  - prepare SQL Server to SQLite migration and validation scripts
- R2 Hot-cold data tiering and archive pipeline
  - define hot retention windows for quote, minute, K-line, news, and analysis history tables
  - move long-range history to Parquet partitions
  - add archive metadata, cleanup jobs, replay path, and retention controls
  - introduce DuckDB queries for local analytics over archive files
- R3 Desktop host self-start integration
  - let the desktop app start and monitor the local backend
  - replace the current start-all development flow with packaged startup behavior
  - add initialization, health checks, port handling, and local exception logging
- R4 Installer and upgrade delivery pipeline
  - unify frontend build, backend publish, and desktop publish
  - define self-contained packaging and WebView2 runtime strategy
  - produce installable artifacts for end users
  - define backup, migration, rollback, and version-upgrade rules

## 分阶段实施清单（ZH）
- R1 数据库提供者抽象与 SQLite 底座
  - 正式定义 provider 选择与本地数据目录配置
  - 移除启动期 SQL Server 方言建表/补列逻辑
  - 让现有核心 schema 和迁移在 SQLite 上完整跑通
  - 准备 SQL Server 到 SQLite 的迁移与校验脚本
- R2 冷热分层与归档链路
  - 为行情、分时、K线、新闻、分析历史定义热数据保留窗口

  - 增加归档元数据、清理任务、回放路径和保留策略
  - 引入 DuckDB 对归档文件做本地分析查询
- R3 桌面宿主自启动整合
  - 让桌面程序负责启动和监控本地后端
  - 用发布态启动流程替代当前 `start-all.bat` 开发流程
  - 增加首次初始化、健康检查、端口处理和本地异常日志
- R4 安装与升级交付链路
  - 统一前端 build、后端 publish、桌面 publish
  - 明确 self-contained 打包和 WebView2 runtime 策略
  - 产出可给终端用户使用的安装包
  - 定义数据备份、迁移、回滚和版本升级规则

## Risks / Issues (EN)
- The current schema initializer approach is the main migration blocker, not the entity model itself.
- WebView2 packaging strategy must be decided early because it affects the final installer shape.
- If large historical data is not physically tiered, backup and upgrade cost will grow quickly even if SQLite remains acceptable at first.

## 风险 / 问题（ZH）
- 当前最大的迁移障碍不是实体模型，而是启动期 schema initializer 对 SQL Server 方言的硬依赖。
- WebView2 的发布与运行时策略需要尽早定，否则会影响最终安装包形态。
- 如果不做冷热物理分层，即使 SQLite 初期可用，随着历史数据增长，备份、升级和恢复成本也会很快恶化。
