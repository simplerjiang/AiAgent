# GOAL-012-R3 Strategy Registry And Multi-Signal Overlay Plan (2026-03-14)

## EN
### Summary
Created the next Dev2 workstream that starts immediately after GOAL-012-R2 closes. The objective is no longer a single-indicator enhancement; it is a full chart-strategy platform built on the current `klinecharts` adapter boundary. The repo already exposes the right seams in `StockCharts.vue`, `useStockChartAdapter.js`, and `klinechartsRegistry.js`, so the next step is to formalize a strategy registry contract and ship grouped button-driven overlays, indicators, and markers without changing the parent component contract.

### Why This Starts Now
- GOAL-012-R2 has already finished the chart-engine replacement and left a working registry boundary.
- The user explicitly requested that all known strategies be included in the plan rather than only `KDJ` or `TD Sequential`.
- The existing UI already has `featureVisibilityByView` and button toggles, which means the strategy system can extend the current model instead of inventing a second control layer.
- This remains fully isolated from Dev1 Step 4.3 because it stays inside `frontend/src/modules/stocks/charting/**`.

### Scope Boundaries
Dev2 may change only:
- `frontend/src/modules/stocks/StockCharts.vue`
- `frontend/src/modules/stocks/charting/**`
- `frontend/package.json`
- required frontend test files

Dev2 must not change:
- backend `TradingPlan` / Step 4.3 trigger engine code
- `StockAgentAnalysisHistory` contract
- stock plan save/edit APIs
- non-chart page business logic

### Required Architecture
1. Introduce a unified strategy registry rather than hardcoding chart branches in `StockCharts.vue`.
2. Each strategy definition must include at least:
   - `id`
   - `label`
   - `category`
   - `kind` (`overlay` / `indicator` / `marker` / `signal`)
   - `supportedViews`
   - `defaultVisible`
   - `requires`
   - `compute(context)`
3. Strategy output must be normalized into a single renderer-facing structure:
   - `indicators`
   - `overlays`
   - `markers`
   - `signals`
   - `legend`
4. Strategy computation and chart rendering must remain decoupled:
   - strategy files compute
   - registry wires availability and visibility
   - `klinechartsRegistry.js` performs concrete chart API mapping

### Strategy Catalog Included In This Plan
1. Trend and baseline overlays:
   - MA5 / MA10 / MA20 / MA60
   - VWAP
   - base line / previous close line
   - BOLL
   - Donchian Channel
2. Momentum and oscillator indicators:
   - MACD
   - KDJ
   - RSI
   - ATR
3. Marker and signal strategies:
   - TD Sequential / 神奇九转
   - MA golden cross / death cross
   - MACD golden cross / death cross
   - KDJ golden cross / death cross
   - breakout / breakdown
   - fake breakout
   - gap up / gap down / gap-fill
   - price-volume divergence
   - VWAP reclaim / VWAP loss
   - opening range breakout (ORB)
4. Existing features that must be absorbed into the same registry:
   - AI resistance/support lines
   - minute base line
   - volume pane visibility

### View Gating Rules
- `minute`: prioritize VWAP, base line, price-volume divergence, VWAP strength/weakness, ORB, intraday crosses.
- `day`: prioritize MA, BOLL, MACD, KDJ, RSI, TD Sequential, breakout signals.
- `month` / `year`: limit initial support to MA, BOLL, MACD, RSI, Donchian.
- Unsupported strategies must not render meaningless disabled buttons.

### Delivery Phases
1. Phase A: strategy registry foundation
   - extract strategy definitions
   - unify per-view visibility state
   - test view filtering and renderer mapping
2. Phase B: baseline indicators
   - MA expansion
   - VWAP normalization
   - BOLL, MACD, KDJ, RSI, ATR, Donchian
3. Phase C: signal markers
   - TD Sequential
   - golden/death cross families
   - breakout / fake breakout
   - gaps and divergence
   - ORB and VWAP strength markers
4. Phase D: interaction hardening
   - grouped buttons/chips by category
   - stable sync across `minute/day/month/year`
   - Browser MCP verification with real toggles

### Testing And Acceptance
1. Unit tests must cover:
   - per-view strategy availability
   - toggle active/inactive classes
   - stable mapping order for representative strategies
   - no duplicate marker emission for TD/cross/breakout signals
2. Required commands:
   - targeted frontend unit tests
   - `npm --prefix frontend run build`
   - Browser MCP validation on backend-served page
3. Browser validation must include:
   - switching `分时图 / 日K图 / 月K图 / 年K图`
   - clicking multiple strategy buttons and waiting for visual state changes
   - checking frontend console and backend logs for new runtime errors

### Risks
- Not every strategy is appropriate for every timeframe; forcing universal display will create noise and wrong interpretation.
- Some advanced signals may eventually need more backend data than current `minuteLines` / `kLines`; if that happens, R3 should stop at the interface boundary and document the missing data prerequisite instead of silently extending scope.
- The chart engine is still on a beta `klinecharts` version, so registry code should isolate strategy logic from engine-specific calls to keep rollback cheap.

## ZH
### 摘要

### 为什么现在开始
- GOAL-012-R2 已经完成图表引擎替换，并留出了可扩展 registry 边界。
- 用户已经明确要求把“已知策略全部纳入计划”，而不是只做 `KDJ` 或 `九转` 的零散增强。
- 现有 UI 已有 `featureVisibilityByView` 和按钮切换模型，可以直接扩展，不需要再发明第二套控制层。
- 本任务仍然可以完全隔离在 `frontend/src/modules/stocks/charting/**` 内，不干扰 Dev1 的 Step 4.3 主线。

### 范围边界
Dev2 允许修改：
- `frontend/src/modules/stocks/StockCharts.vue`
- `frontend/src/modules/stocks/charting/**`
- `frontend/package.json`
- 必要的前端测试文件

Dev2 禁止修改：
- 后端 `TradingPlan` / Step 4.3 触发引擎代码
- `StockAgentAnalysisHistory` contract
- 交易计划保存/编辑 API
- 非图表页业务逻辑

### 必须遵守的架构
1. 引入统一 strategy registry，禁止继续把策略分支硬编码回 `StockCharts.vue`。
2. 每个策略定义至少包含：
   - `id`
   - `label`
   - `category`
   - `kind`（`overlay` / `indicator` / `marker` / `signal`）

   - `defaultVisible`
   - `requires`
   - `compute(context)`
3. 策略输出必须归一化成统一渲染结构：
   - `indicators`
   - `overlays`
   - `markers`
   - `signals`
   - `legend`
4. 策略计算与图表渲染必须继续解耦：
   - strategy 文件负责计算
   - registry 负责可用性与显示开关
   - `klinechartsRegistry.js` 负责映射到具体图表 API

### 视图适配规则
- `minute`：优先 VWAP、基准线、量价背离、VWAP 强弱、ORB、分时金叉类。

- `month` / `year`：初期仅支持 MA、BOLL、MACD、RSI、Donchian 这类中长周期适用指标。
- 不支持的策略不要渲染成无意义的禁用按钮。

   - 代表性策略的映射顺序与输出正确
   - 九转/金叉/突破 marker 不重复落点
2. 必须执行：
   - 前端定向单测
   - `npm --prefix frontend run build`

### 风险
- 不是所有策略都适合所有周期，强行全周期展示只会制造噪音和误判。
- 一部分高级信号未来可能需要超出当前 `minuteLines` / `kLines` 的额外数据；若出现这种情况，R3 应停在接口层并写清数据前提，而不是偷偷扩 scope。
- 当前图表引擎仍是 beta 版 `klinecharts`，因此策略逻辑要尽量隔离于引擎 API，保证后续回退成本可控。
