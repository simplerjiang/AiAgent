# GOAL-018 阶段一执行稿（2026-04-20）

> 状态：讨论版，仅聚焦阶段一；确认后再进入实际开发。

## 本轮一句话目标

这一轮只做一件事：把 **计划 -> 执行 -> 反馈** 做成一个真正能落地使用的最小闭环，重点先做 **交易后标记与反馈**，**不做交易前阻断**。

### 产品原则 / 本轮决策

- **手动优先、AI 可选**：交易计划的主来源必须是用户手动填写、手动确认；AI / 指挥助手 只作为可选起草参考，不能成为默认强依赖。即使用户不信任 LLM，也应该能完整使用 GOAL-018 的核心能力。
- **扩展现有交易日志，不推翻重做**：本轮以增量方式扩展现有 `TradeLogTab.vue`，保留现有持仓总览、风险敞口、交易健康度、AI 教练 / 复盘等已有能力，新能力以执行反馈工作区 / 视图 / 模块的方式接入，不删除旧功能，不破坏已有使用习惯。
- **动作层主表达继续收敛**：动作层的主表达继续固定为 **当前持仓快照** + **今日动作计划**，不再回到 `空仓 / 底仓 / 半仓 / 重仓` 这类抽象仓位标签。
- **计划卡的持仓快照分层口径固定**：
  - **活跃计划 / 未完成计划**：必须在卡内用独立且显眼的 **当前持仓快照** 区块展示并读取实时持仓快照，随最新行情更新当前持仓、浮盈亏、风险敞口等值。
  - **已完成计划**：必须明确切换为 **历史持仓快照（执行时 / 完成时）** 区块，保留回写时的历史值，不再继续刷新实时值，避免破坏复盘事实。
- **交易日志与录入执行也要显式展示快照语境**：`交易日志` 右侧反馈面板必须看得到当前查看的是 **当前持仓快照** 还是 **执行时快照**；`录入执行` 弹窗中的快照必须作为单独、清晰、只读的模块展示，不能埋在说明句里。
- **实时价格能力同时服务“场景状态提示”**：实时价格能力不仅用于刷新持仓快照，也要给预案卡提供一个轻量、显眼的 **场景状态提示区**（小 icon + 文案即可），帮助用户快速判断**当前市场更像处于主场景、备选场景，还是已经命中放弃 / 失效条件**；命中放弃条件时必须切换成明显告警态，不能仍然像正常待执行卡那样展示。
- **场景状态检查按需触发，不做后台持续计算**：阶段一的场景状态检查只在用户主动查看相关页面 / 卡片时触发，默认触发时机包括：打开 `股票信息 -> 当前交易预案` 区块、打开 `交易日志` 中相关预案 / 反馈面板、点击刷新预案卡、进入 `录入执行` 弹窗；如果用户没有打开这些页面 / 卡片，就不需要后端定时器，也不需要后台持续拉取 quote 去刷新场景状态。
- **计划卡回写与交易日志单一事实来源不变**：计划卡回写与交易日志继续共用同一条执行事实，不维护两套独立记录。

## 交易计划 = 场景 + 当前持仓快照下的今日动作预案，而不是单一路径预测

这里说的“交易计划”，在阶段一里更准确的理解应是：**交易预案 = 行情场景层 + 今日动作层**，而不是盘前押注唯一走势的单一路径预测。

也就是说，系统不是只回答“今天可能怎么走”，而是要提前准备成一句更能落地的话：**如果这样走，基于我当前已经记录在系统里的持仓，今天这只票我准备做什么动作。**

这点必须结合现有底座来设计：系统现在已经有真实持仓记录，而且持仓快照就在 `交易日志` / `portfolio snapshot` 里存在，所以阶段一不能再把用户抽象成“空仓 / 底仓 / 半仓 / 重仓”的仓位标签题，而应直接利用**当前持仓快照**来组织预案。

所以阶段一的设计原则要明确成下面这几个点：

- 预案仍然分成两层：
  - **场景层**：描述走势 / 盘口场景，例如回踩承接、修复站回、走弱失效、超预期强
  - **动作层**：描述基于当前持仓快照的**今日计划动作**，例如买入、卖出、做T、持有不动、锁盈减仓、止损退出
- 一张计划卡里允许存在：
  - **主预案**：最优先关注的场景 + 对应今日动作安排
  - **备选预案**：出现不同盘口状态时切换使用的场景 + 对应今日动作安排
  - **放弃条件**：哪些情况出现后今天不做这笔，或已有仓位也不再继续硬扛
- 动作层的主表达，不再是“空仓 / 底仓 / 半仓 / 重仓”这类抽象仓位标签，而是两件更贴近现实的东西：
  - **当前持仓快照**：例如持仓股数、成本价、浮盈亏、可用仓位 / 风险敞口
  - **场景下的计划动作**：例如回踩补仓、冲高减仓、做T卖出、做T买回、继续持有、失效退出
- 计划卡里的持仓快照展示口径进一步固定为：
  - **活跃计划 / 未完成计划** 跟随实时持仓快照刷新，确保盘中查看与执行入口读取的是最新值
  - **已完成计划** 固定展示执行时 / 完成时的历史快照，用于复盘与事实追溯，不再继续更新
- 除了刷新当前持仓快照，**实时价格能力还要驱动预案卡的场景状态提示**：对用户来说，重点不是只看“计划写了什么”，而是知道**当前市场正在靠近哪种预案场景**
- 首版场景状态建议固定为 4 态，文案尽量直白：
  - `👀 待观察`：当前尚未明显进入任一场景
  - `✅ 主场景进行中`：实时价格 / 条件更贴近主场景
  - `🟡 备选场景进行中`：实时价格 / 条件更贴近备选场景
  - `⛔ 放弃条件命中`：已进入失效 / 放弃状态，不建议继续按原计划执行
- 这个场景状态层在阶段一明确只做**轻量提示层**，不是复杂策略引擎：首版优先基于**实时价格、触发价 / 失效价、放弃条件文本、已有计划告警事件（如有）**，给用户一个“当前更接近哪个场景”的提示，而不是宣称机器已经精确理解盘口全貌
- 一旦命中放弃条件 / 失效条件，预案卡必须以明显告警样式呈现，不再继续伪装成普通待执行卡
- 首版仍支持 **轻量动作分支**，但这里的“分支”要收敛为交易动作分支，而不是仓位标签分类；重点是让用户表达“今天在这个场景下准备做什么”，不做复杂策略引擎
- 用户可以在 **开盘后一小时左右** 根据实际走势选择启用哪个预案，并结合系统里当时的持仓快照执行对应动作，而不是要求盘前押单一方向
- 首版只要求把“场景 + 今日动作计划”结构表达清楚、能录入、能回看、能回写，不追求做成很重的流程引擎
- 计划卡回写与交易日志联动逻辑保持不变：**保存执行后自动联动交易日志，且共用同一条执行事实**

这样更贴近用户真实交易方式，尤其更符合 A 股 T+1 的实际语境：**单只股票当日可执行的关键动作通常有限，用户真正需要的不是抽象仓位分层，而是基于当前持仓快照提前写清“今天如果这样走，我准备买、卖、做T、持有还是退出”。**

## 为什么这样做

这轮不继续铺大，只做阶段一，是因为现在最缺的不是“更多统计”，而是 **计划和结果没有真正扣在一起**。

当前问题很明确：

1. 有计划，但执行后很难立刻看出是不是按计划做的。
2. 有交易日志，而且现有基础并不差，尤其 AI 教练分析已经有价值；但它还可以进一步长成更强的纪律反馈台。
3. 有复盘能力，但单笔交易的偏差信息还不够结构化，复盘质量会受影响。
4. 真实交易并不总是盘前押一个方向，更常见的是先准备多种场景，盘中再确认启用哪套预案。
5. 真实交易里用户常常已经有仓位，而且系统里已经有持仓快照，因此最有价值的并不是“预测走势”，而是“明确基于当前持仓，今天在不同场景下该做什么动作”。
6. A 股是 T+1，单只股票日内可执行的关键动作通常只有 1~2 次；用户更需要的是贴着当下持仓做今日动作计划，而不是再套一层抽象的“空仓 / 半仓 / 重仓”标签。
7. 系统既然已经能读取实时价格，阶段一就应该顺手把它用于**场景状态提示**：让用户一眼知道当前更靠近主场景、备选场景还是已命中放弃条件，而不是只停留在“计划文本写过什么”。
8. 这个场景状态提示在阶段一采用**按需检查**模式即可：用户打开预案卡、打开反馈面板、点击刷新、进入录入执行弹窗时检查一次；如果用户没有看这些页面 / 卡片，就不需要后端定时器或后台持续刷新。

所以阶段一的方案刻意收窄：

- 不碰 `情绪轮动`
- 不碰顶层导航
- 不碰 admin 页面
- 不做交易前阻断
- 先把交易后标记和反馈做扎实

这样做的核心逻辑很简单：**先让每一笔交易都能回答“它来自哪条计划、在什么场景下、对应当前持仓做了什么动作、有没有变形、结果怎样、要不要复盘”**。

同时把“计划”收敛成“轻量场景 + 今日动作预案”，更贴近真实交易方式：**盘前准备多种场景，盘中择机确认场景，并根据当前持仓快照执行买 / 卖 / 做T / 持有 / 退出等今日动作，而不是强行预测唯一走势。**

这里再明确一个务实口径：**阶段一的场景状态提示只是轻量提示层，不是复杂策略引擎。** 首版优先基于实时价格、触发价 / 失效价、放弃条件文本、已有计划告警事件（如有）去提示“当前更接近哪个场景”，不宣称系统已经精确理解盘口全貌。

特别是在 A 股 T+1 语境下，这种收敛是必要的：单只股票当日真正有意义的动作窗口并不多，系统更应该帮助用户围绕现有持仓想清楚“今天这一下要怎么做”，而不是把注意力耗在抽象仓位分类上。

而且这个方案**不以用户信任 LLM 为前提**：核心价值来自用户手动计划与执行反馈闭环，AI / 指挥助手 只是在需要时提供可选参考，不是前置条件。

## 对交易过程的具体帮助

### 1. 减少执行变形

当计划卡和执行结果直接关联后，用户会更容易看到：

- 是不是追价了
- 是不是超仓了
- 是不是没有按触发位执行
- 是不是在已有仓位的情况下冲动补仓、冲动砍仓，或者把本来该做T的动作做成了情绪化追单

这会让“偏离计划”变成可见问题，而不是盘后模糊印象。

尤其在 A 股 T+1 交易里，用户经常不是空仓起步，而是已经真实持有这只票；而且单只股票日内通常只有 1~2 次关键动作机会。此时最有价值的不是再听一遍“可能怎么走”，而是提前明确：**如果这样走，基于我现在这笔持仓，今天该买、该卖、该做T、该持有，还是该退出。**

这样能显著减少因为已有仓位而产生的犹豫、冲动补仓、冲动砍仓。

### 2. 提高复盘质量

如果交易后录入时就把偏差标签打好，复盘就不再只剩“这笔赚了/亏了”，而是能直接看到：

- 亏损是市场问题，还是执行问题
- 盈利是否来自纪律内执行
- 哪些错误是重复出现的

复盘从“看结果”升级成“看过程质量”。

### 3. 让计划和结果真正关联起来

很多交易系统的问题不是没有计划，而是计划写在一边，执行和结果记在另一边，最后关联断掉。

本轮要解决的正是这件事：

- 从预案卡进入执行录入
- 录入时展示当前持仓快照，并确认本次执行动作与对应计划动作
- 保存后自动新增 / 更新交易日志记录
- 同一条执行事实同时回写预案卡
- 在交易日志里继续看到反馈

并且这里要明确坚持 **单一事实来源**：计划卡回写与交易日志不是两套独立记录，而是同一条执行数据在两个界面中的不同视图。

这样用户才能形成连续认知，而不是来回切页面靠脑补串联，也避免未来出现双写分叉、状态不一致。

这条闭环默认基于用户自己写和确认的计划成立；AI 只是在需要时辅助起草或提示，不接管主流程。

### 4. 提升纪律训练的可操作性

阶段一不做硬阻断，但先把偏差数据沉淀出来，后面才有资格决定要不要做更强的纪律约束。

也就是说，这一轮是在为以后更强的纪律机制打地基，但先不抢跑。

### 5. 更贴近真实交易台使用方式

专业交易台不是让人填很多表，而是让人快速看到：

- 我原来准备了哪些场景
- 当前市场更像正在靠近哪种场景
- 我当前系统记录的持仓快照是什么
- 我最终启用了哪个场景、对应哪条今日动作计划
- 我实际上怎么做的
- 我哪里做偏了
- 下一次该修什么

本轮 UI 方案就是围绕这几个问题来组织信息。

这也更贴近 A 股实际使用方式：用户一天对单只股票通常只有少数几次关键决策，不需要系统反复追问“你现在算半仓还是重仓”，更需要系统直接把当前持仓摆出来，再让用户确认“今天这一笔准备怎么做”。

## 本轮验收口径

阶段一完成后，至少要满足以下验收口径：

1. 在 `股票信息 -> 当前交易预案` 中，用户能从某条预案直接进入“录入执行”。
2. 活跃计划 / 未完成计划的计划卡，必须以独立显眼区块展示 **当前持仓快照**；已执行 / 已完成计划的计划卡，必须明确展示 **历史持仓快照（执行时 / 完成时）**。
3. 保存执行后，计划卡上能看到执行回写、偏差摘要、反馈状态。
4. 活跃预案卡能显示当前场景状态提示；命中放弃条件时有明显告警样式，不再与普通待执行卡混同。
5. 活跃预案卡、`交易日志` 反馈面板、`录入执行` 弹窗都能在**打开时 / 主动查看时**拿到场景状态；该能力不依赖后台持续轮询。
6. 保存执行后会自动新增 / 更新 `交易日志` 中对应的执行记录，且与计划卡回写共用同一条执行事实。
7. `交易日志` 能清楚区分：`计划内`、`偏离计划`、`无计划`。
8. `交易日志` 主界面能查看单笔交易的反馈面板，而不是只看流水文本；反馈面板里还必须看得到当前是 **当前持仓快照（活跃态）** 还是 **执行时快照（历史态）**，并能回看**执行前场景状态 / 预案状态快照**。
9. 活跃计划 / 未完成计划展示的持仓快照应基于实时持仓与最新行情刷新；已完成计划保留执行时 / 完成时的历史快照，不继续刷新当前实时值。
10. “录入执行”弹窗能完成基础成交录入 + 交易后标记，并支持选择主场景 / 备选场景 / 放弃交易；其中 **当前场景状态** 与 **当前持仓快照（只读）** 都必须作为独立区块展示，不得退化为附带说明。
11. `交易日志` 的定位明确为交易后录入/复盘，不被改成交易前计划页。
12. 本轮改动不影响：
   - `情绪轮动` 页面
   - 顶层导航
   - admin 页面
13. `交易日志` 现有持仓总览、风险敞口、交易健康度、AI 教练 / 复盘等能力在新增执行反馈工作区后仍可继续使用。
14. 阶段一不引入新的后端轮询 worker，也不为了场景状态提示新增持续运行的后台任务。
15. 整体链路能跑通一条最小闭环：
  - 新建或已有计划
  - 盘前准备主场景 + 备选场景 + 放弃条件，并写清对应今日动作计划
  - 活跃预案卡在用户打开 / 刷新时按需显示当前场景状态提示，必要时切换到 `⛔ 放弃条件命中`
  - 开盘后一小时左右选择启用哪个场景，并确认当前持仓快照下准备执行的动作
  - 录入执行
  - 在 `交易日志` / 反馈面板打开相关记录时按需检查并回看执行时的场景状态
  - 查看偏差与反馈
  - 进入复盘

## 待用户确认的问题

只需要确认 4 个点，确认后即可把阶段一继续压成开发任务：

1. `交易日志` 右侧反馈面板是否接受“教练提示”这种直白表达，还是要更克制一些？
2. “偏差标签”首版是否就按这 4 类起步：`追价`、`超仓`、`提前执行`、`无计划交易`？
3. 计划卡上的执行回写，首版是只显示“最近一次执行”，还是要显示“多次执行汇总”？
4. “需要复盘”是否作为保存时可勾选的强字段，还是先做成轻量标签？
