# 交易日志-计划管理工作区 UI/UX 设计讨论稿

日期：2026-04-21  
适用页面：`frontend/src/modules/stocks/TradeLogTab.vue`

## 目标与问题定义

当前 `交易日志` 页面已经具备：

- 交易记录列表
- 交易复盘工作区
- 从交易计划跳转进来的执行录入链路

但它还缺一个**跨股票、统一查看所有交易计划的工作区**。现在用户想看“我当前到底有哪些计划、哪些快到触发、哪些已经失效、哪些已经有执行回写、当前价格大概到哪了”，需要在多个入口之间来回切换，成本偏高。

本次设计要解决的问题是：

1. 在 `交易日志` 页面提供一个统一的计划管理入口。
2. 用户能按状态、方向、关键词快速筛选/搜索计划。
3. 用户能快速看到计划最关键的几项：**股票、当前价、触发价、状态、最新执行摘要、是否有告警**。
4. 计划与交易记录、右侧复盘工作区形成联动，帮助用户做“交易后查看 / 联动复盘”。

> 这不是把 `交易日志` 做成第二个 `股票信息`。这里的定位是：**交易后的计划跟踪与复盘工作区**，不是盘前深度编辑计划的主战场。

## 为什么要放在 `交易日志` 页面

### 1. 这里天然是“执行与复盘”场景

`TradeLogTab.vue` 当前已经承接：

- 交易记录浏览
- 快速录入执行
- 右侧复盘工作区
- 从计划跳入执行录入

也就是说，计划一旦进入“执行、偏差、回写、复盘”阶段，最自然的落点就是 `交易日志`，而不是继续停留在单只股票详情页。

### 2. 用户在这里看的是“计划是否被执行、是否偏离、后续怎么复盘”

放在这里，计划不再只是静态文本，而是和以下信息绑定：

- 已发生的交易
- 当前持仓/回写状态
- 复盘关注点
- 后续是否还需要跟踪

这比在 `股票信息` 页里单独看计划，更符合“交易后管理”的心智。

### 3. 可以避免把 `股票信息` 页面继续做重

`股票信息` 页更适合：

- 单只股票研究
- 新建/编辑计划
- 看详细上下文

`交易日志` 页更适合：

- 横向扫全部计划
- 发现异常
- 关联查看执行
- 快速进入复盘

这样的职责拆分更清楚。

## 设计原则

### 1. MVP 优先，先做统一查看与联动

MVP 阶段优先复用现有能力：

- 计划列表：`GET /api/stocks/plans?take=50`
- 计划预警：`GET /api/stocks/plans/alerts?take=20`
- 计划标准化/格式化：`stockInfoTabTradingPlans.js`
- 现有视觉模块：`StockTradingPlanBoard.vue`、`StockTradingPlanSection.vue`

先解决“看得全、找得到、能联动”，不急着把复杂编辑能力搬过来。

### 2. 信息优先级必须清晰

在 `交易日志` 页里，计划卡片/行项只保留最值得先看的信息：

1. 股票 + 计划摘要
2. 当前价 / 触发价
3. 状态
4. 最新执行摘要
5. 告警
6. 快捷操作

像长篇分析、完整风险条款、完整表单字段，不应该默认铺开。

### 3. 与交易日志联动，而不是并排互不相干

点中计划后，页面应该立即出现反馈：

- 右侧复盘工作区切换到该计划上下文
- 下方交易记录列表高亮或筛出关联交易
- 用户能快速判断：这个计划有没有执行、执行得怎么样、现在还要不要继续盯

### 4. 不做第二个 `股票信息`

本页不推荐默认提供完整计划编辑表单；即使后续允许快速编辑，也应是轻量动作或跳转动作，而不是把 `StockTradingPlanModal.vue` 原样塞进主工作区变成大表单页面。

## 推荐布局方案

### 主推荐方案：在左侧主内容区增加“计划管理工作区”，保留右侧复盘工作区

推荐把新的计划管理工作区插入到当前左侧主列中，位置在：

- 交易工具栏 / 汇总卡片之后
- 交易记录列表之前

这样页面会形成明确结构：

1. 顶部：当前已有的交易日志工具栏与汇总
2. 中部：新增“计划管理工作区”
3. 下部：交易记录列表
4. 右侧：现有“复盘工作区”保持不动，作为联动详情区

这样做的好处：

- 不打断当前交易日志主流程
- 计划与交易记录上下相邻，便于联动
- 右侧详情区可直接复用，避免再造一个计划详情面板
- 用户能先横向扫计划，再往下看对应交易

### 推荐的视觉形态

不建议直接照搬股票页的大卡片；推荐使用**紧凑行卡（row-card）**，保留现有计划卡的徽标、pill、告警样式，但压缩为更适合列表扫描的密度。

建议每条计划默认一行半到两行信息，支持展开查看次要信息。

### 备选方案（不推荐优先）

可选做法是把左侧主区改成二级 Tab：

- `交易记录`
- `计划管理`

但这会带来两个问题：

1. 计划和交易记录被切开，联动感变弱。
2. 用户切到计划后，看不到下面的交易列表，不利于复盘串联。

因此只作为备选，不作为首推方案。

## 信息结构

### 1. 顶部工具栏

计划管理工作区顶部建议包含：

- 标题：`计划管理工作区`
- 辅助说明：`用于交易后查看、筛选和联动复盘，不承载深度编辑`
- 搜索框：支持前端本地搜索 `symbol / name / analysisSummary`
- 筛选项（MVP）
  - 状态：全部 / 待执行 / 已触发 / 需复核 / 已失效
  - 方向：全部 / 做多 / 做空
  - 仅看告警：开关
- 重置按钮
- 刷新按钮（刷新计划列表与预警）

> 搜索和 richer filter 在 MVP 阶段先做前端本地过滤即可，避免先加新接口。

### 2. 计划列表主体

每条计划建议展示以下字段，按优先级排列：

#### 第一层：一眼判断是否值得点开

- 股票代码 + 名称
- 计划状态
- 当前价
- 触发价
- 当前场景/市场阶段（简写）
- 最新执行摘要（如：已执行 1 次 / 最近动作为加仓执行）

#### 第二层：帮助理解上下文

- 计划摘要（单行截断）
- 主/备场景
- 建立时市场上下文
- 当前市场上下文
- 告警摘要（如有）

#### 第三层：快捷操作

- `选中复盘`：把该计划送入右侧复盘工作区
- `查看关联交易`：让下方交易列表进入关联过滤/高亮状态
- `查看股票`：跳回股票信息页查看详细计划与研究上下文
- `录入执行`：进入已有执行录入链路（如本计划状态允许）

### 3. 当前价展示建议

当前价建议放在计划项的第一视觉层，但需要明确定位：

- 它是**辅助判断计划是否接近触发/失效**，不是逐秒行情主战场。
- MVP 允许使用 plans 列表现有返回字段中的当前参考价/当前价能力。
- 如果某条计划没有可用当前价，则显示 `--`，不要伪装成实时行情。
- 若接近触发价或失效价，可加弱提醒色块，但避免做成行情软件式闪烁。

### 4. 下方交易列表的联动信息

当用户选中某个计划后，交易列表区域建议增加一条轻量状态条：

- `当前查看：宁德时代计划 #12`
- `已高亮关联交易`
- `清除联动`

这样用户能知道为什么下面的交易列表排序/高亮发生了变化。

## 交互说明

### 1. 选中计划后的联动

用户点击某条计划的主区域或 `选中复盘` 后：

1. 该计划行进入选中态。
2. 右侧 `复盘工作区` 切换为该计划上下文。
3. 页面调用已有计划执行上下文能力，展示：
   - 当前计划
   - 最近执行摘要
   - 当前场景状态
   - 当前持仓快照
4. 下方交易记录列表进入“计划联动模式”：
   - 关联该计划的交易优先显示或高亮
   - 无关联计划的交易保持可见，但弱化

### 2. 点击“查看关联交易”

推荐区分两种动作：

- `选中复盘`：驱动右侧工作区
- `查看关联交易`：驱动下方交易列表的过滤/定位

两者可以同时发生，但文案上要明确，避免用户以为只是一个“查看详情”按钮。

### 3. 点击“查看股票”

跳转到对应股票详情页，用于：

- 深度编辑计划
- 查看完整研究上下文
- 做更复杂的计划修改

这能明确边界：

- `交易日志` 页负责统一查看与联动复盘
- `股票信息` 页负责单票深度操作

### 4. 点击“录入执行”

若计划状态允许录入执行，建议直接复用现有执行录入流程，让用户从计划管理工作区快速进入执行回写，而不是重新设计一套入口。

## 空态 / 加载态 / 错误态

### 空态

#### 全局无计划

文案建议：

- `暂无交易计划，先去股票信息页建立计划，后续会在这里统一跟踪。`

操作建议：

- 提供 `去股票信息查看` 或 `返回交易记录` 的轻操作

#### 有计划但筛选后为空

文案建议：

- `当前筛选条件下没有匹配的计划。`
- `试试清空关键词或放宽状态筛选。`

### 加载态

建议使用 5-6 条骨架屏行，保留列表结构，不要只显示一句“加载中...”，否则会让中部区域显得空。

### 错误态

建议在工作区内部显示：

- 错误文案：`交易计划加载失败`
- 次要说明：`可先查看交易记录，或点击重试刷新计划区`
- 按钮：`重试`

不要让计划区错误影响整页交易日志的可用性。

## MVP 范围

### 本次建议纳入 MVP 的内容

1. 在 `TradeLogTab.vue` 左侧主列增加统一的计划管理工作区。
2. 拉取全局计划列表与全局计划预警。
3. 支持前端本地搜索与基础筛选。
4. 列表中展示：股票、状态、当前价、触发价、计划摘要、执行摘要、告警。
5. 点击计划后驱动右侧复盘工作区与下方交易列表联动。
6. 提供轻量快捷操作：`选中复盘 / 查看关联交易 / 查看股票 / 录入执行`。

### 暂不做项

1. 不在 `交易日志` 页内提供完整计划编辑表单主视图。
2. 不做复杂的服务端多条件搜索、排序、分页。
3. 不做逐秒实时行情刷新。
4. 不做批量编辑、批量归档、批量状态变更。
5. 不做高级看板切换（卡片/表格双视图、多列拖拽、自定义列）。

## 风格与排版建议

- 工作区外层卡片延续当前页面的浅色卡片体系。
- 行项间距建议使用 `12px ~ 16px`，避免过于拥挤。
- 一条计划默认不超过 3 个主色：
  - 蓝：中性信息 / 计划标签
  - 绿：正向状态 / 已触发 / 顺势
  - 橙或红：告警 / 接近失效 / 需复核
- 当前价与触发价建议使用等宽数字样式，便于快速扫视。
- 摘要文案建议单行或双行截断，避免卡片高度失控。

## 需要 PM / 用户拍板的关键决策

1. **点中计划后，交易列表应默认“自动过滤到关联交易”，还是只做高亮不做过滤？**
   - 自动过滤更聚焦
   - 只高亮更不容易让用户迷路

2. **MVP 是否允许在 `交易日志` 页直接使用“录入执行”按钮？**
   - 允许：闭环更快
   - 不允许：边界更清楚，只保留查看与跳转

3. **计划区默认展示的数据范围是否先固定为最近 50 条？**
   - 固定 50 条实现快
   - 若用户更关心“全部未失效计划”，后续可能要改成状态优先而非时间优先

4. **当前价字段在 MVP 中是否明确视为“参考当前价”，而非实时行情？**
   - 如果是，就需要在文案上避免给用户“秒级更新”的预期

## 结论

本方案建议把 `交易日志` 页升级为一个**以复盘联动为核心的计划管理工作区**：

- 计划统一查看在这里完成
- 深度编辑仍回到 `股票信息`
- 右侧复盘区继续做详情承接
- 下方交易列表继续做执行记录承接

这样既复用了现有计划能力，也避免把页面做成第二个单票研究页，整体更符合“交易后跟踪、执行回写、联动复盘”的场景定位。
